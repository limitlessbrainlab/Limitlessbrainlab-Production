2. Application Overview

Neurosense is an AI-powered neurological assessment platform that enables clinicians to analyze patient brain data, generate clinical reports, and track treatment outcomes using advanced machine learning.

Core User Flow (as implemented in the UI)

  • Patients register via email or clinic referral and access their brain wellness data and training protocols.
  • Clinicians upload EEG files or raw brain data, which are processed by Google Gemini API to generate AI-powered clinical reports.
  • Clinics manage patient rosters, subscriptions, and team members. They assign brain training protocols based on assessment results.
  • Super Admins monitor system health, manage clinic accounts, track usage analytics, and handle billing.
  • Reports are automatically generated, exported as PDFs, and can be emailed to patients or shared via the platform.

Main Tabs & User Interfaces

Patient Dashboard

Purpose: Patients track their brain wellness reports, complete training protocols, and manage medical history.
Key Pages: Dashboard home, My Reports, Brain Training, Progress Tracking, Medical History

Clinic Portal

Purpose: Clinicians manage patient rosters, generate reports, track subscriptions, and view team analytics.
Key Pages: Patient List, Report Generator, Subscription Management, Team Settings, Clinic Analytics

Admin Dashboard

Purpose: Super Admins manage clinics, monitor system health, view usage analytics, and handle disputes.
Key Pages: User Management, Clinic Overview, System Health, Analytics & Reports, Payment Reconciliation

Core Features

  • QEEG Report Generation: Upload EEG files → AI generates detailed clinical reports → Export as PDF
  • Patient Management: Register, track medical history, assign training protocols, monitor progress
  • Brain Training: Personalized protocols based on assessment results
  • Subscription Management: Clinic subscriptions with Razorpay billing
  • Real-time Notifications: Report status updates, payment confirmations, and clinical alerts
  • Secure Data Access: Row-level security ensures patients only see their own data

6. Database Schema

The database is hosted on Supabase (PostgreSQL). All tables have Row-Level Security (RLS) enabled to ensure users can only access their own data.

Core Tables

users

Stores all user accounts (patients, clinicians, admins)

- id (UUID, primary key) - email (TEXT, unique) - password_hash (TEXT) - role (TEXT: 'patient', 'clinic', 'admin', 'super_admin') - clinic_id (UUID, foreign key to clinics) - created_at (TIMESTAMP) - updated_at (TIMESTAMP)

patients

Extended patient information and medical history

- id (UUID, primary key) - user_id (UUID, foreign key to users) - first_name (TEXT) - last_name (TEXT) - date_of_birth (DATE) - medical_history (JSONB) - referral_source (TEXT) - clinic_id (UUID, foreign key to clinics) - created_at (TIMESTAMP)

clinics

Clinic and healthcare provider information

- id (UUID, primary key) - name (TEXT, unique) - contact_email (TEXT) - phone (TEXT) - address (TEXT) - subscription_tier (TEXT: 'basic', 'pro', 'enterprise') - created_at (TIMESTAMP)

qeeg_reports

Generated QEEG analysis reports

- id (UUID, primary key) - patient_id (UUID, foreign key to patients) - clinic_id (UUID, foreign key to clinics) - file_name (TEXT) - ai_analysis (JSONB) — Gemini API response - pdf_url (TEXT) — Supabase Storage path - status (TEXT: 'processing', 'completed', 'failed') - created_at (TIMESTAMP)

subscriptions

Clinic and patient subscription records

- id (UUID, primary key) - clinic_id (UUID, foreign key to clinics) - tier (TEXT: 'basic', 'pro', 'enterprise') - status (TEXT: 'active', 'cancelled', 'paused') - razorpay_subscription_id (TEXT) - billing_amount (NUMERIC) - renewal_date (DATE) - created_at (TIMESTAMP)

brain_training_protocols

Personalized training programs assigned to patients

- id (UUID, primary key) - patient_id (UUID, foreign key to patients) - protocol_name (TEXT) - description (TEXT) - exercises (JSONB) - duration_days (INTEGER) - assigned_date (TIMESTAMP) - completed_date (TIMESTAMP)

Key Security Features

  • Row-Level Security (RLS): Patients can only view their own records
  • Clinic Isolation: Clinicians can only manage their own clinic's patients
  • Admin Oversight: Super admins have read access to all data for auditing
  • Parameterized Queries: All database queries use parameterized statements to prevent SQL injection
  • Encryption at Rest: Supabase handles encryption of data stored in PostgreSQL
  • Encryption in Transit: All connections are HTTPS/TLS encrypted

8. Authentication & User Roles

Authentication Flow

Login Process

  1. User enters email and password
  2. Frontend calls POST /api/auth/login
  3. Backend verifies credentials and generates JWT
  4. JWT stored in browser localStorage with 24-hour expiry
  5. Subsequent requests include Authorization: Bearer {token} header
  6. Middleware verifies JWT on every protected route

Token Management

  • JWT Expiry: 24 hours
  • Refresh Endpoint: POST /api/auth/refresh-token
  • Storage: Browser localStorage (accessible to XSS scripts — mitigated with CSP headers)
  • Secure Flag: In production, tokens can be stored in httpOnly cookies for XSS protection

User Roles & Permissions

Role Description Permissions
Patient End user accessing their health data View own reports, track training, update medical history, manage profile
Clinic Clinician or clinic staff member Upload EEG files, generate reports, manage patient roster, view clinic analytics
Admin Clinic administrator with billing access Manage subscriptions, handle payments, view clinic staff, update clinic settings
Super Admin Platform administrator with full access Manage all users, monitor system health, view all analytics, handle disputes

RBAC Middleware (Role-Based Access Control)

Every protected endpoint enforces role checks via middleware:

// Example: Only admins can access user management router.get('/admin/users', authMiddleware, rbac('admin'), getUserList); // Only clinic staff can generate reports router.post('/qeeg/process', authMiddleware, rbac('clinic'), processQEEG); // Patients can only access their own data (enforced by RLS) router.get('/patients/:id', authMiddleware, getPatient);

Security Best Practices

  • Password Hashing: Bcrypt with salt cost 12
  • Session Management: Stateless JWT (no server-side sessions)
  • CSRF Protection: SameSite cookies for CSRF prevention
  • Rate Limiting: Login attempts limited to 5/minute per IP
  • 2FA (Optional): Can be integrated via TOTP
  • Audit Logging: All sensitive actions logged with user ID and timestamp

Purpose: Analyze EEG files and generate clinical reports

  • Use Case: Process raw EEG data → medical-grade analysis
  • Rate Limit: ~20 requests/day (configurable in environment)

  • Config Variable: GEMINI_API_KEY (must be in render.yaml with sync:false)

Purpose: PostgreSQL database, user authentication, file storage

  • Database: All user data, patient records, subscriptions, reports
  • Authentication: User login/signup via Supabase Auth
  • Storage: PDF reports stored in patient-reports bucket
  • Row-Level Security: Policies enforce data isolation by user/clinic

14. Maintenance & Troubleshooting

Common Issues & Solutions

401 Unauthorized Errors

Symptom: API returns 401 Unauthorized

Root Cause: Missing or expired JWT token

Solution:

  1. Check browser localStorage for auth_token
  2. Verify token isn't expired (24-hour TTL)
  3. Call POST /api/auth/refresh-token to refresh
  4. Re-login if token cannot be refreshed

QEEG Report Generation Fails

Symptom: Report processing stuck or times out

Root Causes:

  • Gemini API quota exhausted (20 requests/day limit)
  • EEG file format invalid (must be PDF or image)
  • Network timeout on Gemini API call

Solution:

  1. Check server/logs/app.log for Gemini error messages
  2. Verify GEMINI_REQUEST_DELAY_MS=2000 in render.yaml (not higher)
  3. Verify GEMINI_API_KEY is set in Render dashboard
  4. Wait until next day if daily quota exhausted
  5. Try with different EEG file format

PDF Download Issues

Symptom: "Failed to download PDF" error on frontend

Root Causes:

  • PDF file not found in Supabase Storage
  • Missing Authorization header in download request
  • Mixed content error (HTTP/HTTPS mismatch)

Solution:

  1. Verify PDF was generated and stored (check Supabase Storage)
  2. Ensure Bearer token is included in request header
  3. Use HTTPS URLs in production (not HTTP)
  4. Check browser console for specific error messages

Database Connection Errors

Symptom: "Connection refused" or "FATAL: all connections used" error

Solution:

  1. Check Supabase dashboard for connection pool status
  2. Verify SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY are correct
  3. Restart backend server (Render will auto-restart on redeploy)
  4. Contact Supabase support if quota exceeded

Email Not Sending

Symptom: Patient doesn't receive report email

Solution:

  1. Check server/logs/app.log for email errors
  2. Verify Gmail credentials: EMAIL_USER, EMAIL_PASS
  3. Ensure Gmail account has "Less secure app access" enabled (deprecated) or use app-specific password
  4. Check spam folder
  5. Verify recipient email address is valid

Maintenance Tasks

Weekly

  • Check error logs: server/logs/app.log
  • Monitor API response times via observability dashboard
  • Verify all integrations are functioning

Monthly

  • Review security logs for suspicious activity
  • Test backup and restore procedures
  • Check database query performance and optimize if needed
  • Verify SSL certificates haven't expired
  • Review Gemini API usage and quota

Quarterly

  • Disaster recovery drill
